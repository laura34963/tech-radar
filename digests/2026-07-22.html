GHSA-gcfj-64vw-6mp9: axios

axios Node HTTP adapter 在 interceptor 複製 config 後，可能沿用繼承來的 proxy 設定，導致請求走向非預期的代理。

Why it matters

axios 已列於你們的 npm 相依套件中，且 Node.js 後端（含 Next.js API route、SSR 呼叫）常以它發外部請求；一旦流量被導向惡意 proxy，等同開了 SSRF 與中間人的破口。

Recommended action

展開詳細

stack: axios 2026-07-20

GHSA-pjwm-pj3p-43mv: axios

axios 的 shouldBypassProxy 無法辨識 IPv4-mapped IPv6 位址，使 NO_PROXY 白名單可被繞過（CVE-2025-62718 的補丁不完整）。

Why it matters

在 Kubernetes/Docker 雙協定網路環境中，服務常以 IPv6-mapped 形式解析內部位址，NO_PROXY 被繞過會讓原本受信任的東西向流量意外離開叢集邊界。

Recommended action

更新至同時修補此不完整補丁的 axios 版本，並實測以 ::ffff: 前綴的內網位址確實走直連而非 proxy。

展開詳細

shouldBypassProxy 比對 NO_PROXY 時未正規化 ::ffff:x.x.x.x 這類 IPv4-mapped IPv6 表示法，導致原本應直連的內網位址仍被送往 proxy。這是先前 CVE-2025-62718 修補未涵蓋的殘留路徑，攻擊者可藉此讓針對內部服務的請求外流或被攔截。

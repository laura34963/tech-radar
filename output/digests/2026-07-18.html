stack: axios 2026-07-17

GHSA-pjwm-pj3p-43mv: axios

axios NO_PROXY bypass via unrecognized IPv4-mapped IPv6 addresses (incomplete fix for CVE-2025-62718).

Read more

axios's shouldBypassProxy fails to normalize IPv4-mapped IPv6 addresses (e.g. ::ffff:127.0.0.1) when matching hosts against NO_PROXY, so a host meant to bypass the proxy is still routed through it. This is an incomplete fix for CVE-2025-62718, where the original patch did not cover the IPv4-mapped IPv6 representation. The result is that proxy-bypass policy can be silently defeated for affected address forms.

Why it matters

This stack depends directly on axios in the npm ecosystem, so any service or frontend/next.js code issuing outbound HTTP through axios is exposed. NO_PROXY misrouting can leak internal traffic through an untrusted proxy or undermine egress controls, which is a meaningful risk for server-side requests in Rails/Next.js backends that call internal hosts.

Recommended action

Upgrade axios to the version that patches GHSA-pjwm-pj3p-43mv (the follow-up to CVE-2025-62718) and run npm audit to confirm the advisory is cleared across the lockfile.
